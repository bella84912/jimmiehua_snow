# 安全防護設定

## 我的任務

我做兩件事，第一件**必做**、第二件**選用**：

1. **（必做）黑名單**：把危險指令加進 Claude Code 的黑名單，避免 AI 在執行任務時不小心：
   - 刪除重要檔案
   - 強制覆蓋 git
   - 關機、重開機
   - 破壞磁碟
2. **（選用）白名單／自動允許**：把「常用又安全」的動作設成自動放行，讓用戶不用每次都被跳出來問「要不要允許」。危險的動作仍然保留要問或直接擋掉——所以開了白名單也不會變得不安全。

> 黑名單做完後，再問用戶要不要順手做第二件（見 Step 5）。

---

## 執行流程

### Step 1：開場說明

「我來幫你設定 Claude Code 的安全防護 🛡️

接下來會把以下危險指令加入黑名單：
- `rm -rf` 等強制刪除指令
- `git push --force`、`git reset --hard`
- `sudo` 系列指令
- `shutdown`、`reboot`、`mkfs` 等系統指令

設定好之後，Claude 遇到這些指令會直接拒絕執行，保護你的電腦。

開始嗎？」

等用戶確認。

---

### Step 2：讀取現有設定

讀取 `~/.claude/settings.json`：
- 如果檔案不存在 → 建立新的，從空 JSON 開始
- 如果已存在 → 保留原有設定，只在 `permissions.deny` 區塊加入新規則

---

### Step 3：加入黑名單規則

在 `~/.claude/settings.json` 的 `permissions.deny` 陣列加入以下指令（如果已存在則跳過）：

```json
"Bash(rm -rf /)",
"Bash(rm -rf /*)",
"Bash(rm -rf ~)",
"Bash(rm -rf ~/*)",
"Bash(rm -rf .)",
"Bash(rm -rf ./*)",
"Bash(rm -fr *)",
"Bash(rm -Rf *)",
"Bash(git reset --hard *)",
"Bash(git push --force *)",
"Bash(git push -f *)",
"Bash(git branch -D *)",
"Bash(git clean -f *)",
"Bash(sudo *)",
"Bash(shutdown *)",
"Bash(reboot *)",
"Bash(diskutil erase *)",
"Bash(dd *)",
"Bash(mkfs *)"
```

**重要**：使用 Read + Edit/Write 來更新 `settings.json`，**不要**用 bash 指令處理（避免 JSON 格式跑掉）。

---

### Step 4：驗收

告訴用戶：

「✅ 安全防護已就位

加入了 19 條黑名單規則，分三類：
- 🗑️ 強制刪除（rm -rf / git clean 等）
- 💥 Git 危險操作（force push、hard reset 等）
- ⚙️ 系統指令（sudo、shutdown、磁碟操作等）

之後 Claude Code 如果遇到這些指令，會直接拒絕執行——你的電腦安全了。

設定檔位置：`~/.claude/settings.json`，之後想看或調整都可以打開檢查。」

接著主動問用戶要不要做第二件（選用）：

> 「另外問你——你平常用團隊的時候，Claude 是不是常常跳出來問『要不要允許』，按到有點煩？我可以幫你把**常用又安全**的動作設成自動放行，危險的還是會擋。要的話我幫你設，2 分鐘 👉」

用戶說要 → 進 Step 5；不要 → 就到這裡結束。

---

### Step 5（選用）：自動允許白名單——不用每次手動按「允許」

> 🎯 先用白話講清楚這是什麼，再動手。

**為什麼會一直跳「允許」？** Claude Code 為了安全，預設「動到你電腦的動作」都會先問過你（讀檔、改檔、跑指令、發文、拉數據都算）。好處是安全，壞處是用熟之後很煩。

**白名單的原理：** 把「你信得過、又每天在用」的動作寫進 `permissions.allow`，這些就不再問、直接做；其他沒列進去的照樣會問；Step 3 黑名單裡的危險動作則永遠擋掉。**三者的優先順序是「擋 > 問 > 放行」，所以開白名單不會把危險的也放出去。**

#### Step 5-1：讓用戶選「放行到哪個程度」（給選項，不要他自己想）

> 「你想放行到哪個程度？選一個：
>
> **A. 輕度（最安全，推薦先用這個）**　只自動放行「唯讀」動作：讀你的檔案、上網查資料。完全不會改到任何東西。
>
> **B. 中度（最實用）**　在 A 之上，再放行團隊每天要做的：改你的品牌資料與草稿、發文 / 拉廣告數據這類動作。改檔、發文不用再一直按允許，但刪除、系統指令還是擋著。
>
> **C. 我自己挑**　你告訴我有哪些操作老是跳允許很煩，我針對那幾項加就好。」

#### Step 5-2：依選擇寫進 `permissions.allow`

用 Read 讀 `~/.claude/settings.json`，在 `permissions.allow` 陣列加入對應規則（保留原有內容，已存在的跳過）。

**A. 輕度（唯讀）：**

```json
"Read",
"WebFetch",
"WebSearch",
"Bash(git status)",
"Bash(git log:*)",
"Bash(git diff:*)"
```

**B. 中度（= A 再加下面這些）：**

```json
"Edit",
"Write",
"mcp__*"
```

> 說明給用戶聽：
> - `Edit` / `Write`＝讓 Claude 改檔、存檔（例如更新品牌資料、存貼文草稿）不用每次問。
> - `mcp__*`＝讓已經接好的工具（發 FB/IG、拉 Meta 數據、寫 Google 日曆）直接執行。**只有你自己接過的工具才會生效**，沒接的不會憑空冒出來。
> - 就算開了中度，Step 3 黑名單的 `rm -rf`、`sudo`、force push 等危險動作**仍然會被擋下來**——這就是為什麼可以放心。

**C. 自己挑：** 依用戶講的痛點，加對應規則（例如老是卡在發 IG → 加對應的 `mcp__` 工具名；老是卡在跑某個指令 → 加 `Bash(該指令:*)`）。不確定規則怎麼寫，就先幫他加最接近的，並說明之後可隨時調整。

> ⚠️ **不要做「全部自動允許」**（例如把整個 `Bash(*)` 或所謂的「跳過所有權限」打開）。那等於讓 AI 做任何事都不問你，風險太高。用戶若主動要求，要先清楚告知風險、確認他真的了解，再決定。

#### Step 5-3：驗收（白名單）

告訴用戶：

> 「✅ 自動允許設好了（你選的是 [輕度／中度／自訂]）
>
> 之後這些動作 Claude 會直接做、不再問你：[條列剛加的白名單]
>
> 但放心，[Step 3 的危險指令] 還是會被擋；沒列到的動作也照樣會先問過你。
>
> 要關掉或調整，打開 `~/.claude/settings.json` 把 `permissions.allow` 裡的那幾行刪掉就好。改完**完全關閉 Claude Code 再重開**才會生效。」

## 行為規則

1. 動手修改 `settings.json` **之前**先確認用戶同意
2. 修改時要**保留原有設定**，只在 `permissions.deny`（黑名單）或 `permissions.allow`（白名單）增加規則，不要覆蓋其他區塊
3. 如果 `settings.json` 格式有問題，先告訴用戶、不要強行寫入
4. 完成後**列出加了哪些規則**，讓用戶清楚自己被保護了什麼、又放行了什麼
5. **白名單是選用的**——一定先做完黑名單，再問用戶要不要做；用選項讓他選程度，預設推薦「輕度」
6. **絕不主動開「全部自動允許」**（`Bash(*)` 或跳過所有權限）；用戶硬要才做，且必須先講清楚風險
7. 使用 Read + Edit/Write 更新 `settings.json`，**不要**用 bash 處理（避免 JSON 格式跑掉）；改完提醒用戶**完全關閉 Claude Code 再重開**才生效
